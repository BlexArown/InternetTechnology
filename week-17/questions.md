### Какие критерии вы использовали при разделении монолита на микросервисы?

Сервисы разделялись по бизнес-ответственности.

В проекте `shipments-s07` выделены:

- `shipments-svc-s07` — основной сервис отправлений;
- `tracking-service` — сервис отслеживания tracking-кодов;
- `notification-service` — сервис уведомлений.

Такое разделение выбрано потому, что каждый сервис отвечает за отдельную область логики. Это уменьшает связанность компонентов и позволяет развивать, тестировать и масштабировать сервисы независимо.

### Почему вы выбрали именно этот стек технологий?

Для реализации выбран Python и FastAPI, потому что FastAPI позволяет быстро писать REST API, имеет встроенную документацию и хорошо подходит для учебных микросервисов.

Docker и Docker Compose выбраны для контейнеризации и локального запуска всей системы одной командой.

REST используется для внешнего API, потому что он прост, понятен и удобен для клиента.

Для внутреннего взаимодействия между сервисами в текущей реализации используется HTTP, но в архитектуре предусмотрена возможность перехода на gRPC для более строгих контрактов и эффективного межсервисного обмена.

### Как ваши сервисы справляются с ошибками?

В проекте используется базовая graceful degradation.

Например, если `notification-service` недоступен, создание shipment не ломается полностью: основной сервис перехватывает ошибку и продолжает работу.

Если `tracking-service` недоступен, сервис возвращает ошибку `503 Service Unavailable`, показывая, что зависимый сервис временно недоступен.

В production-версии можно улучшить обработку ошибок через:

- retry;
- circuit breaker;
- очереди сообщений;
- таймауты;
- централизованное логирование.

### Как организован деплой и обновление вашей системы без простоя?

В текущей учебной реализации деплой организован через Docker Compose:

docker compose up --build

Для production-версии предполагается Kubernetes Deployment с Rolling Update.

Rolling Update позволяет обновлять сервис постепенно:

- сначала создаются новые Pod-ы;
- затем старые Pod-ы удаляются;
- сервис продолжает принимать трафик во время обновления.

Также можно использовать readinessProbe, чтобы Kubernetes отправлял трафик только на готовые контейнеры.

### Что было самым сложным в интеграции всех компонентов вместе?

Самым сложным было связать несколько сервисов так, чтобы они корректно общались внутри Docker Compose сети.

Нужно было правильно настроить:

- имена сервисов;
- порты;
- healthcheck;
- depends_on;
- Gateway;
- маршрутизацию запросов между контейнерами.

Также важно было сделать так, чтобы проект запускался одной командой и не требовал ручного запуска каждого сервиса отдельно.
