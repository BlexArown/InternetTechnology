## Что такое Server Streaming? Приведите пример использования.

Server Streaming — это тип RPC в gRPC, при котором клиент отправляет один запрос, а сервер возвращает поток сообщений (несколько ответов подряд).

В отличие от обычного запроса (Unary RPC), где ответ один, здесь соединение остается открытым, и сервер может постепенно отправлять данные.

Пример использования:
- стрим логов сервера;
- получение новостной ленты;
- обновления котировок (биржа);
- поток телеметрии или датчиков.

Например, клиент подписывается на обновления:
```proto
rpc SubscribeSessions (SubscribeSessionsRequest) returns (stream SessionUpdate);```

Сервер после этого начинает отправлять новые данные по мере их появления.

## Чем Server Streaming отличается от просто получения большого списка (List) в одном ответе?

Основное отличие — в способе передачи данных и задержке получения информации.

Server Streaming:
- данные приходят по частям (потоком);
- клиент может начать обработку сразу;
- не нужно ждать, пока соберется весь набор данных;
- подходит для "живых" обновлений.

Большой список (List):
- сервер формирует весь ответ целиком;
- клиент получает данные только после полной обработки;
- выше задержка (latency);
- может занимать больше памяти.

Идея:
- List — "подожди и получи всё сразу";
- Streaming — "получай по мере готовности".

## Какие метрики производительности существуют (Latency, Throughput, CPU Usage)?

Основные метрики:

### Latency (задержка)
Время от отправки запроса до получения ответа (или первого байта).
Чем меньше — тем быстрее отклик системы.

### Throughput (пропускная способность)
Сколько запросов система может обработать за единицу времени (например, RPS — requests per second).
Чем больше — тем лучше система справляется с нагрузкой.

### CPU Usage (использование процессора)
Насколько сильно нагружается процессор при обработке запросов.
Высокое значение может означать неэффективную обработку данных.

В контексте gRPC:
- Latency обычно ниже из-за бинарного формата;
- Throughput выше благодаря HTTP/2;
- CPU используется эффективнее, чем при JSON.

## Почему важно переиспользовать канал (Channel) в gRPC клиенте при замерах производительности?

Создание канала (Channel) — это дорогостоящая операция:
- устанавливается соединение;
- выполняется настройка HTTP/2;
- происходит согласование параметров.

Если создавать новый канал для каждого запроса:
- результаты замеров будут искажены;
- появятся лишние накладные расходы;
- производительность будет казаться хуже, чем есть на самом деле.

Правильный подход:
- создать канал один раз;
- использовать его для всех запросов.

Это позволяет измерить именно производительность метода, а не стоимость установки соединения.

## В каких сценариях REST может оказаться быстрее или удобнее gRPC?

REST может быть предпочтительнее в следующих случаях:

- публичные API (доступ из браузера);
- простые запросы без сложных структур;
- когда важна читаемость (JSON);
- удобство тестирования (curl, Postman);
- отсутствие необходимости в генерации кода;
- интеграция с фронтендом без дополнительных библиотек.

Также REST может быть быстрее:
- при очень простых данных (разница минимальна);
- когда нет выгоды от HTTP/2;
- при использовании кэширования (HTTP cache).

Идея:
REST — проще и универсальнее,  
gRPC — эффективнее при нагрузке и сложных структурах.

## Что такое "Head-of-Line Blocking" и как HTTP/2 (и gRPC) решает эту проблему?

Head-of-Line Blocking — это ситуация, когда один медленный запрос блокирует выполнение следующих.

В HTTP/1.1:
- запросы обрабатываются последовательно;
- если один "завис", остальные ждут.

Это создает задержки и снижает производительность.

HTTP/2 решает эту проблему за счет:
- мультиплексирования (несколько запросов одновременно в одном соединении);
- разделения данных на независимые потоки;
- параллельной обработки.

gRPC работает поверх HTTP/2, поэтому:
- запросы не блокируют друг друга;
- повышается пропускная способность;
- уменьшается задержка.

Итог:
HTTP/2 устраняет "узкое место", позволяя обрабатывать несколько запросов одновременно.
