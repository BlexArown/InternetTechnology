# Ответы на вопросы по лабораторной работе (gRPC)

## 1. Зачем нужен файл `.proto`? Почему мы не пишем классы сообщений сразу на Python?

Файл `.proto` используется для описания контракта между клиентом и сервером. В нем задаются:
- сервисы (service),
- методы (rpc),
- структуры сообщений (message).

На основе `.proto` автоматически генерируется код для разных языков (Python, Java, Go и др.).

Если писать классы сразу на Python:
- они будут работать только для Python,
- не будет единого контракта,
- сложнее обеспечить совместимость между разными сервисами.

Таким образом, `.proto` — это единый источник истины (contract-first подход).

---

## 2. Что такое Unary RPC? Чем он концептуально похож на REST запрос?

Unary RPC — это тип вызова, при котором:
- клиент отправляет один запрос,
- сервер возвращает один ответ.

Это самый простой тип взаимодействия в gRPC.

Он похож на REST-запрос:
- один HTTP-запрос → один HTTP-ответ,
- есть входные данные и результат.

Разница в том, что:
- REST работает поверх HTTP + JSON,
- gRPC использует HTTP/2 и бинарный формат (Protobuf).

---

## 3. Почему gRPC обычно эффективнее REST (JSON) по процессору и трафику?

Основные причины:

### 1. Бинарный формат (Protobuf)
- компактнее JSON,
- быстрее сериализация и десериализация,
- меньше нагрузка на CPU.

### 2. Меньше размер сообщений
- JSON содержит текстовые ключи,
- Protobuf использует числовые теги → меньше данных передается.

### 3. HTTP/2
- мультиплексирование (несколько запросов в одном соединении),
- меньше накладных расходов,
- лучше работа с задержками.

В итоге:
- меньше трафика,
- быстрее обработка,
- лучше производительность.

## 4. Что такое "обратная совместимость" в Protobuf и зачем нужны номера полей (тэги, например `= 1`)?

Обратная совместимость — это возможность:
- изменять структуру сообщений,
- не ломая старые версии клиентов и серверов.

В Protobuf каждое поле имеет номер (тэг):

```proto
string id = 1;```

Этот номер:

- используется при сериализации,
- важнее, чем имя поля.

Почему это важно:

- можно добавлять новые поля (с новыми номерами),
- старые клиенты их просто игнорируют,
- новые клиенты могут работать со старыми данными.

Главное правило:  
**нельзя менять номера существующих полей.**

## 5. Что такое Stub (стаб) на клиенте и Servicer на сервере?

### Stub (клиент)

Это сгенерированный класс, который:

- позволяет вызывать удалённые методы как обычные функции,
- скрывает сетевую работу (RPC, сериализацию и т.д.).

Пример:

```python
response = stub.GetBooking(request)```

Клиент думает, что вызывает локальный метод.

---

### Servicer (сервер)

Это базовый класс, который:

- генерируется из `.proto`,
- от него наследуется серверная реализация.

Разработчик:

- реализует методы сервиса,
- пишет бизнес-логику.

## 6. В каких случаях вы бы выбрали REST, а в каких — gRPC?

### REST подходит, если:

- API публичный (например, используется в браузере),
- нужна простота и читаемость,
- важно легко тестировать (Postman, curl),
- используется JSON и HTTP/1.1,
- нет строгих требований к производительности.

---

### gRPC подходит, если:

- важна высокая производительность,
- используется микросервисная архитектура,
- требуется строгая типизация,
- взаимодействуют сервисы на разных языках,
- нужны стримы или минимальные задержки,
- важна компактность передаваемых данных.

---

## Вывод

REST — проще, понятнее и универсальнее.
gRPC — быстрее, строже и эффективнее.

На практике часто используют комбинацию:

- внешний API → REST
- внутренние сервисы → gRPC
