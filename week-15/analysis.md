# Анализ производительности comments-s07

## 1. Информация о сервисе

- project_code: comments-s07
- service: comments-svc-s07
- endpoint: http://localhost:8258/comments

## 2. Условия тестирования

- инструмент: wrk
- длительность: 30 секунд
- тестируемый endpoint: /comments
- измеряемые метрики:
  - latency (avg, P50, P99)
  - throughput (RPS)

## 3. Результаты тестирования

| Concurrency | RPS (Requests/sec) | Avg Latency | P50 Latency | P99 Latency |
|------------:|-------------------:|------------:|------------:|------------:|
| 1           | 2186 req/s         | 0.50 ms     | 0.47 ms     | 0.85 ms     |
| 10          | 2795 req/s         | 3.92 ms     | 3.79 ms     | 6.38 ms     |
| 100         | 2685 req/s         | 37.44 ms    | 36.45 ms    | 55.97 ms    |

## 4. Анализ latency и throughput

При concurrency = 1 latency минимальная (~0.5 ms), но throughput ограничен (~2186 RPS).

При увеличении concurrency до 10:
- throughput значительно увеличивается (~2795 RPS)
- latency увеличивается, но остаётся в пределах нескольких миллисекунд

При concurrency = 100:
- throughput почти не растёт (~2685 RPS)
- latency резко увеличивается (~37 ms среднее, ~56 ms P99)

## 5. Точка насыщения

Точка насыщения для comments-s07 находится примерно на уровне concurrency = 10.

После этого:
- throughput перестаёт расти
- latency начинает расти резко

Это означает, что система достигает предела по CPU/IO и не может эффективно обрабатывать больше запросов.

## 6. Выводы

1. Сервис comments-s07 хорошо масштабируется при малой и средней нагрузке.
2. При увеличении concurrency растёт throughput, но также увеличивается latency.
3. После достижения точки насыщения дальнейшее увеличение нагрузки не даёт прироста RPS.
4. Основной показатель деградации — рост P99 latency, что влияет на пользовательский опыт.
5. Для улучшения производительности можно:
   - увеличить количество worker-процессов
   - добавить кэширование
   - масштабировать сервис горизонтально
