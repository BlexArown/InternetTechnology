### Почему мы создаем Deployment, а не сразу Pod?

Pod — это временная сущность. Если он упал или был удалён, он не восстанавливается сам.

Deployment — это управляющий объект, который следит за Pod-ами:
- поддерживает нужное количество реплик
- автоматически пересоздаёт Pod при падении
- позволяет обновлять приложение (rolling update)

То есть Deployment — это "контроллер", который обеспечивает стабильность, а Pod — просто исполняемый объект.

### В чем разница между Liveness и Readiness пробами? Что произойдет, если провалится каждая из них?

Liveness probe проверяет, "жив ли контейнер".
Если она проваливается — Kubernetes перезапускает контейнер.

Readiness probe проверяет, "готов ли контейнер принимать трафик".
Если она проваливается — Pod исключается из Service и не получает запросы, но не перезапускается.

Итого:
- Liveness fail → перезапуск контейнера
- Readiness fail → временно не участвует в обработке запросов

### Что такое "requests" и "limits" для CPU и памяти? Почему важно их указывать?

Requests — это минимальные ресурсы, которые контейнер гарантированно получит.
Limits — это максимальные ресурсы, которые контейнер может использовать.

Если контейнер превышает:
- CPU limit → его ограничивают (throttling)
- memory limit → контейнер убивается (OOM Kill)

Важно указывать:
- чтобы планировщик правильно размещал Pod-ы
- чтобы один контейнер не "съел" весь сервер
- для стабильной работы кластера

### Как Service находит свои Pod-ы? (Labels & Selectors)

Service использует selector, чтобы выбрать Pod-ы по labels.

В Deployment:
labels:
  app: likes-app

В Service:
selector:
  app: likes-app

Service ищет все Pod-ы с подходящими labels и направляет на них трафик.

### Какой тип сервиса используется для внутреннего общения, а какой — для внешнего доступа?

ClusterIP — используется для внутреннего общения внутри кластера (по умолчанию).

NodePort — открывает порт на каждой ноде, позволяет доступ снаружи через IP ноды.

LoadBalancer — создаёт внешний балансировщик (обычно в облаке) и даёт публичный IP.

Итого:
- внутренний доступ → ClusterIP
- внешний доступ → NodePort или LoadBalancer

### Что такое "Rolling Update" в контексте Деплоймента?

Rolling Update — это стратегия обновления, при которой Pod-ы обновляются постепенно, а не все сразу.

Kubernetes:
- создаёт новые Pod-ы с новой версией
- постепенно удаляет старые
- следит, чтобы приложение оставалось доступным

Это позволяет обновлять приложение без простоя (zero downtime).
