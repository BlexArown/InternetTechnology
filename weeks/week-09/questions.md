## Зачем нужен Signaling сервер, если WebRTC — это P2P технология?

Хотя WebRTC обеспечивает прямое (Peer-to-Peer) соединение между браузерами, на начальном этапе им нужно "познакомиться" и обменяться служебной информацией.

Signaling сервер нужен для:
- передачи SDP (описания соединения);
- передачи ICE-кандидатов;
- координации установления соединения.

Сам сервер не участвует в передаче данных после установления соединения. Он используется только на этапе настройки.

Идея:
- Signaling = этап договорённости;
- WebRTC = дальнейшая прямая передача данных.

## В чем разница между STUN и TURN серверами?

STUN и TURN используются для установления соединения между клиентами за NAT.

### STUN (Session Traversal Utilities for NAT)

- помогает определить внешний IP и порт клиента;
- позволяет попытаться установить прямое соединение (P2P);
- не передает сами данные.

### TURN (Traversal Using Relays around NAT)

- используется, если прямое соединение невозможно;
- работает как ретранслятор (сервер-посредник);
- весь трафик проходит через него.

Разница:
- STUN → помогает соединиться напрямую;
- TURN → передает данные вместо прямого соединения.

## Что такое ICE Candidate?

ICE Candidate — это возможный способ (адрес), по которому можно установить соединение с клиентом.

Он содержит:
- IP-адрес;
- порт;
- тип (локальный, внешний, через TURN).

Процесс:
1. Клиент собирает список кандидатов (ICE gathering);
2. Отправляет их другому клиенту через Signaling;
3. Оба клиента пробуют соединиться по разным вариантам;
4. Выбирается рабочий путь.

Таким образом, ICE Candidate — это "вариант маршрута" для соединения.

## Почему WebRTC обычно использует UDP, а не TCP?

WebRTC чаще использует UDP, потому что он лучше подходит для real-time коммуникации.

Причины:

- UDP быстрее: нет подтверждений доставки (ACK);
- ниже задержка (latency);
- нет повторной отправки пакетов, которая замедляет поток;
- подходит для видео, аудио и чата в реальном времени.

TCP:
- гарантирует доставку и порядок;
- но может задерживать данные при потере пакетов.

Идея:
лучше получить данные быстро, даже с потерями, чем идеально, но с задержкой.

## Что такое SDP (Session Description Protocol) и какая информация в нем содержится?

SDP (Session Description Protocol) — это формат описания параметров соединения между двумя участниками WebRTC.

Он передается через Signaling сервер.

SDP содержит:

- поддерживаемые кодеки (видео/аудио);
- тип соединения (offer или answer);
- параметры передачи данных;
- сетевую информацию (частично);
- настройки медиа-потоков или data channel.

SDP используется для:
- согласования параметров соединения;
- понимания, как именно клиенты будут общаться.

Проще говоря:
SDP — это "описание условий соединения", по которым договариваются два клиента.
