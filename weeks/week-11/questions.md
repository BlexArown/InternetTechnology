### Зачем создавать пользовательскую сеть (bridge network) в Docker Compose, если есть дефолтная?

По умолчанию Docker создает сеть для compose-проекта, и сервисы могут общаться друг с другом. Однако явное создание пользовательской сети дает больше контроля.

Преимущества пользовательской сети:
- Явное управление: можно задать имя сети и явно указать, какие сервисы к ней подключены;
- Изоляция: можно разделять сервисы по разным сетям (например, frontend и backend);
- Предсказуемость: удобно при отладке и масштабировании, когда инфраструктура становится сложнее;
- Совместимость: можно подключать внешние сервисы или несколько compose-проектов к одной сети.

Таким образом, пользовательская сеть делает конфигурацию более прозрачной и управляемой.

### В чем разница между build и image в описании сервиса?

Параметр build используется, когда нужно собрать образ из Dockerfile.

Пример:
build:
  context: ./app

В этом случае Docker будет собирать образ локально перед запуском контейнера.

Параметр image используется, когда образ уже существует (локально или в Docker Hub/registry).

Пример:
image: nginx:alpine

Разница:
- build → сборка образа из исходников;
- image → использование уже готового образа.

Иногда их комбинируют: сначала build, затем задают имя image для тегирования.

### Что делает depends_on? Гарантирует ли он, что приложение в зависимом контейнере полностью инициализировалось?

depends_on задает порядок запуска контейнеров.

Например:
depends_on:
  - db

Это означает, что контейнер db будет запущен раньше, чем текущий сервис.

Однако depends_on не гарантирует, что приложение внутри контейнера полностью готово к работе. Он гарантирует только, что контейнер создан и запущен.

Например:
- база данных может быть запущена, но еще не принимать соединения;
- сервис может быть запущен, но не готов обрабатывать запросы.

Для реальной проверки готовности используется healthcheck.

### Как работают healthcheck-и и как использовать их в depends_on?

Healthcheck — это механизм проверки состояния контейнера. Docker периодически выполняет указанную команду внутри контейнера.

Пример:
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
  interval: 10s
  timeout: 3s
  retries: 3

Если команда выполняется успешно — контейнер считается healthy.
Если несколько раз подряд неудачно — контейнер становится unhealthy.

В связке с depends_on можно использовать:
depends_on:
  db:
    condition: service_healthy

Это означает:
- запуск текущего сервиса произойдет только после того, как db станет healthy.

Таким образом, healthcheck позволяет дождаться реальной готовности сервиса, а не просто его запуска.

### Как передать переменные окружения в контейнер через Compose (минимум 2 способа)?

Способ 1: через environment

services:
  app:
    environment:
      - DB_HOST=db
      - DB_PORT=5432

Или в формате ключ-значение:

environment:
  DB_HOST: db
  DB_PORT: 5432

Способ 2: через env_file

services:
  app:
    env_file:
      - .env

В файле .env:

DB_HOST=db
DB_PORT=5432

Также можно комбинировать оба способа.

Таким образом, Compose позволяет удобно передавать конфигурацию в контейнер без изменения кода.

### Почему Docker Compose редко используют в продакшене (хотя это возможно)?

Docker Compose удобен для локальной разработки, но имеет ограничения для продакшена.

Основные причины:
- Нет автоматического масштабирования (в отличие от Kubernetes);
- Нет продвинутого управления отказами (failover);
- Ограниченные возможности балансировки нагрузки;
- Нет встроенного управления кластером (одна машина или ручная настройка);
- Слабая поддержка rolling updates и zero-downtime деплоя;
- Ограниченный контроль над ресурсами и распределением контейнеров.

Поэтому в продакшене чаще используют Kubernetes или другие системы оркестрации.

Однако Compose подходит для:
- локальной разработки;
- небольших проектов;
- тестовых стендов.
