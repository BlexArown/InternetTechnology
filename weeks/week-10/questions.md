### Что такое слой в Docker и почему порядок команд в Dockerfile важен для кэширования?

Слой (layer) в Docker — это неизменяемый уровень файловой системы, который создается при выполнении каждой инструкции в Dockerfile (например, FROM, RUN, COPY).

Каждый слой кэшируется. Это означает, что если Docker уже выполнял эту инструкцию ранее и она не изменилась, он может использовать готовый слой вместо повторного выполнения.

Порядок команд важен, потому что Docker проверяет кэш сверху вниз. Если изменить одну из инструкций, все последующие слои пересобираются заново.

Например:
- если сначала копировать весь проект, а потом устанавливать зависимости — при любом изменении кода зависимости будут устанавливаться заново;
- если сначала копировать requirements.txt, установить зависимости, а потом копировать код — зависимости будут кэшироваться и не переустанавливаться при изменении кода.

Таким образом, правильный порядок команд ускоряет сборку и уменьшает нагрузку.

### В чем суть Multi-stage build? Какие проблемы он решает?

Multi-stage build — это подход, при котором в Dockerfile используется несколько этапов сборки (несколько инструкций FROM).

На первом этапе (builder) выполняются тяжелые операции:
- установка зависимостей,
- компиляция,
- сборка проекта.

На втором этапе (runtime) используется чистый образ, в который копируются только необходимые результаты (например, установленные библиотеки и код приложения).

Это решает несколько проблем:
- уменьшает размер итогового образа (не попадают компиляторы, временные файлы и кэши);
- повышает безопасность (в контейнере нет лишних инструментов);
- делает контейнер более “чистым” и предсказуемым.

В результате вместо большого образа получается компактный и оптимизированный.

### Зачем нужен файл .dockerignore?

Файл .dockerignore используется для исключения файлов и папок из контекста сборки Docker.

Когда выполняется команда docker build, Docker отправляет в демон все файлы из текущей директории. Без .dockerignore туда попадет всё: .git, виртуальные окружения, кэш и т.д.

.dockerignore позволяет:
- уменьшить размер контекста сборки;
- ускорить сборку;
- избежать попадания лишних или чувствительных файлов в образ;
- уменьшить размер итогового контейнера.

Обычно в него добавляют:
- .venv
- __pycache__
- *.pyc
- .git
- временные файлы и тесты

### Чем отличаются инструкции CMD и ENTRYPOINT?

CMD и ENTRYPOINT задают команду, которая выполняется при запуске контейнера.

Основное отличие:
- CMD можно переопределить при запуске контейнера;
- ENTRYPOINT задает фиксированную команду, которая всегда выполняется.

Пример:
- CMD ["python", "main.py"] — можно заменить при запуске;
- ENTRYPOINT ["python"] + CMD ["main.py"] — python всегда будет выполнен, а main.py можно заменить аргументом.

Таким образом:
- CMD используется для задания команды по умолчанию;
- ENTRYPOINT — для жестко заданного поведения контейнера.

Часто их используют вместе.

### Почему рекомендуется использовать COPY вместо ADD?

Инструкция COPY используется только для копирования файлов и директорий из контекста сборки в образ.

Инструкция ADD делает больше:
- может распаковывать архивы;
- может загружать файлы по URL.

Из-за этого ADD имеет скрытое поведение, которое может привести к неожиданным результатам.

Поэтому рекомендуется использовать COPY, потому что:
- он более предсказуемый;
- делает только одно действие;
- улучшает читаемость Dockerfile;
- снижает риск ошибок.

ADD используют только тогда, когда действительно нужны его дополнительные возможности.

### Почему приложение в контейнере должно слушать 0.0.0.0, а не 127.0.0.1?

Адрес 127.0.0.1 означает “локальный интерфейс внутри контейнера”.  
Если приложение слушает только его, оно будет доступно только внутри контейнера.

Адрес 0.0.0.0 означает “слушать на всех сетевых интерфейсах”.

Docker пробрасывает порт с хоста в контейнер, но для этого приложение должно принимать соединения извне контейнера. Это возможно только если оно слушает 0.0.0.0.

Если использовать 127.0.0.1, то:
- контейнер запустится;
- но снаружи (через localhost:порт) к нему подключиться нельзя.

Поэтому в Docker-контейнерах всегда используют 0.0.0.0.
