### В чем разница между Continuous Integration, Continuous Delivery и Continuous Deployment?

Continuous Integration (CI) — непрерывная интеграция:
разработчики часто вливают изменения в основную ветку, а система автоматически:
- собирает проект,
- запускает тесты,
- проверяет качество кода.

Continuous Delivery (CD) — непрерывная доставка:
после успешного CI код автоматически готовится к релизу (собирается артефакт, Docker-образ и т.д.),  
но выкладка в прод происходит вручную.

Continuous Deployment — непрерывный деплой:
каждое изменение, прошедшее CI, автоматически разворачивается в продакшене без участия человека.

Итого:
- CI → проверка кода
- Delivery → готовность к релизу
- Deployment → автоматический релиз

### Почему тесты должны запускаться до сборки Docker образа (или хотя бы до его пуша)?

Потому что тесты — это самый быстрый способ поймать ошибки.

Если сначала собрать и запушить Docker-образ:
- можно отправить в registry нерабочую версию;
- другие разработчики или сервисы начнут использовать сломанный образ;
- увеличивается время и ресурсы на сборку.

Правильный порядок:
1. lint
2. тесты
3. build
4. publish

Это экономит ресурсы и защищает систему от нерабочих версий.

### Зачем нужно кэширование зависимостей в CI пайплайне?

Без кэширования зависимости (pip, npm и т.д.) скачиваются заново при каждом запуске CI.

Это приводит к:
- долгим сборкам;
- лишней нагрузке на сеть;
- замедлению разработки.

Кэширование позволяет:
- сохранять уже скачанные пакеты;
- повторно использовать их в следующих сборках;
- ускорять pipeline в разы.

Например, cache: pip в GitHub Actions сохраняет установленные Python-зависимости.

### Что такое "Secret" в контексте CI/CD? Как безопасно передать пароль от Docker Hub в пайплайн?

Secret — это защищённая переменная (пароль, токен, ключ API), которая не хранится в коде.

В CI/CD секреты:
- хранятся в настройках (например, GitHub Secrets);
- не попадают в репозиторий;
- подставляются во время выполнения pipeline.

Для Docker Hub:
- создаётся секрет (например, DOCKER_PASSWORD);
- используется в pipeline через переменные окружения;
- передаётся в docker login.

Это защищает конфиденциальные данные от утечек.

### Что такое "Matrix Build" (матричная сборка)?

Matrix Build — это запуск одного и того же pipeline с разными параметрами.

Например:
- разные версии Python (3.9, 3.10, 3.11);
- разные ОС (Linux, Windows);
- разные конфигурации.

CI запускает несколько задач параллельно.

Это позволяет:
- проверить совместимость;
- быстрее находить ошибки;
- тестировать разные окружения одновременно.

### Почему нельзя хранить конфиги продуктовой среды в репозитории кода (или можно, но осторожно)?

Потому что конфиги часто содержат чувствительные данные:
- пароли,
- токены,
- адреса сервисов.

Если хранить их в репозитории:
- есть риск утечки (особенно в публичных репозиториях);
- сложно менять конфигурацию без изменения кода;
- невозможно безопасно разделять dev/stage/prod.

Правильный подход:
- использовать Secrets или переменные окружения;
- хранить конфиги отдельно;
- при необходимости — использовать зашифрованные файлы.

Иногда конфиги можно хранить в репозитории, но без секретов и с осторожностью.
