### Зачем нужен Helm? Какую проблему он решает?

Helm — это пакетный менеджер для Kubernetes.

Он решает проблему дублирования и сложности YAML-манифестов:
- вместо копирования одних и тех же Deployment/Service файлов для разных окружений (dev/stage/prod),
- используется один шаблон (Chart) и разные значения (Values).

Преимущества Helm:
- уменьшает копипасту;
- упрощает изменение конфигурации (например, версии образа);
- позволяет переиспользовать шаблоны;
- делает инфраструктуру более управляемой.

Таким образом, Helm превращает Kubernetes YAML из “кучи файлов” в параметризованный шаблон.

### Как работает приоритет значений (Values)? Кто победит: values.yaml или флаг --set?

В Helm значения могут задаваться из нескольких источников:

1. values.yaml (дефолтные значения)
2. values-*.yaml (overrides)
3. флаг --set в командной строке

Приоритет (от слабого к сильному):
values.yaml < values-*.yaml < --set

То есть:
- значения из values-dev.yaml переопределяют values.yaml;
- значения из --set переопределяют всё остальное.

Победитель: --set имеет самый высокий приоритет.

### Что такое функция include и _helpers.tpl? Зачем они нужны?

include — это функция Helm, которая позволяет вставлять один шаблон в другой.

_helpers.tpl — это специальный файл, где хранятся переиспользуемые шаблоны (например, имена ресурсов, метки и т.д.).

Пример:
{{ include "mychart.fullname" . }}

Зачем это нужно:
- избегать дублирования кода;
- централизовать логику (например, формирование имени ресурсов);
- сделать шаблоны более читаемыми и поддерживаемыми.

Это аналог “функций” или “макросов” в шаблонизации.

### Почему в шаблонах Helm так важно следить за отступами и зачем нужна функция indent или nindent?

YAML чувствителен к отступам — неправильные пробелы ломают структуру файла.

Helm-шаблоны просто подставляют текст, не проверяя корректность YAML.

Функции:
- indent — добавляет отступы;
- nindent — добавляет отступы + перевод строки.

Пример:
{{ .Values.resources | nindent 10 }}

Зачем это нужно:
- сохранить правильную структуру YAML;
- избежать ошибок при рендере;
- сделать шаблон валидным после подстановки значений.

Без правильных отступов Helm может сгенерировать невалидный YAML.

### Как откатить неудачный релиз с помощью Helm?

Helm хранит историю релизов.

Чтобы посмотреть историю:
helm history <release-name>

Чтобы откатиться:
helm rollback <release-name> <revision>

Пример:
helm rollback my-release 1

Это восстановит предыдущую версию Deployment/Service.

Таким образом можно быстро откатить изменения без ручного редактирования YAML.

### Можно ли использовать Helm только для генерации YAML, без установки в кластер?

Да, можно.

Команда:
helm template <release-name> ./chart

Она:
- рендерит шаблоны;
- подставляет значения;
- выводит готовый YAML.

Этот YAML можно:
- посмотреть;
- сохранить в файл;
- применить через kubectl.

Таким образом Helm можно использовать как генератор конфигурации, без установки в Kubernetes.
